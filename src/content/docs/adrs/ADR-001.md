---
title: "ADR-001: Adopt Event-Driven Architecture for GreenGrant"
description: "An architectural decision record describing the choice of a Event-Driven Architecture for the GreenGrant system."
---

- Status: active


## In the context of
- establishing a microservices architecture with independently deployable services
- the GreenGrant nationwide chocolate inventory platform, part of the Zamunda Digital Renaissance (ZDR)
- a distributed system where services must communicate without introducing tight coupling
- the need to react to domain events such as inventory updates, product normalization, stock changes, and future purchasing actions
- increasing integration requirements with external manufacturers and partner systems

---

## Facing the need for
- asynchronous communication between microservices to avoid tight coupling and cascading failures
- reliable propagation of state changes across services (e.g., ingestion → normalization → catalog)
- the ability to support multiple consumers of the same data without modifying producers
- improved system resilience under partial failures or high load
- near real-time processing of events such as inventory updates and notifications
- scalable integration with external systems that may produce or consume data at different rates

---

## We decided for
- an Event-Driven Architecture (EDA) as the primary communication pattern between services, consisting of:
  - services publishing domain events when significant state changes occur
  - services subscribing to relevant events and reacting independently
  - the use of an event broker (e.g., Apache Kafka or RabbitMQ) to decouple - producers and consumers
  - adoption of publish/subscribe patterns rather than point-to-point messaging
  - event schema definition and versioning to ensure compatibility over time
  - asynchronous communication as the default, with synchronous REST used only where necessary (e.g., queries)

---

## Achieving
- loose coupling between services, allowing independent evolution and deployment (modularity)
- improved resilience, as services can continue operating even if others are temporarily unavailable
- scalable event distribution, enabling multiple consumers (e.g., analytics, notifications) without impacting producers
- real-time or near real-time data propagation across the system
- extensibility, allowing new features to be added by introducing new event consumers rather than modifying existing services
- better alignment with domain-driven design through explicit domain events

---

## Accepting that (we will have)
- eventual consistency between services instead of immediate consistency
- increased complexity in understanding and tracing event flows across the system
- the need for robust observability, including centralized logging, metrics, and distributed tracing
- challenges in debugging asynchronous interactions
- responsibility for managing event schemas, compatibility, and evolution
- the need for idempotent consumers and retry mechanisms to handle duplicate or failed event processing
- operational overhead in managing and maintaining the event infrastructure
