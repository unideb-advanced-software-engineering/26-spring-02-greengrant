---
title: "ADR-004: Adoption of PostgreSQL for Data Persistence in GreenGrant"
description: "An architectural decision record describing the choice of PostgreSQL as the primary relational database for the GreenGrant system."
---

* Status: Active

## In the context of

* establishing a microservices architecture with decentralized data ownership  
* adopting an Event-Driven Architecture (EDA) with eventual consistency  
* selecting RabbitMQ as the event broker for asynchronous communication  
* the GreenGrant platform requiring reliable, structured storage for inventory, catalog, and transactional data  
* the need for strong data integrity within individual services  

---

## Facing the need for

* a robust and reliable database system for service-level data persistence  
* strong consistency and transactional guarantees within a service boundary  
* support for relational data models (e.g., inventory, products, suppliers)  
* flexibility to handle moderately complex queries and relationships  
* compatibility with widely used backend frameworks and tools  
* long-term maintainability with a mature and well-supported ecosystem  

---

## We decided for

* using PostgreSQL as the primary database technology for microservices, with:

  * each service owning its own PostgreSQL database/schema  
  * ACID-compliant transactions for strong consistency within service boundaries  
  * support for relational modeling and constraints (foreign keys, indexes)  
  * optional use of JSON/JSONB for semi-structured data where appropriate  
  * integration with service frameworks (e.g., ORM tools or direct SQL access)  

---

## Achieving

* strong data integrity and consistency within each microservice (reliability)  
* clear data ownership aligned with microservice boundaries (modularity)  
* flexibility to model both structured and semi-structured data (adaptability)  
* powerful querying capabilities for complex domain logic (capability)  
* stability and long-term support from a mature open-source ecosystem (sustainability)  

---

## Accepting that (we will have)

* the need to manage multiple database instances (one per service)  
* challenges in maintaining data consistency across services (eventual consistency)  
* additional complexity in handling distributed transactions (avoiding 2PC, using patterns like Saga)  
* potential need for performance tuning as data volume grows  
* operational overhead for backups, scaling, and monitoring
