---
title: "ADR-005: Adoption of NoSQL Database for Specialized Use Cases in GreenGrant"
description: "An architectural decision record describing the use of a NoSQL database to complement PostgreSQL for specific scalability and flexibility requirements."
---

* Status: Active

## In the context of

* establishing a microservices architecture with decentralized data ownership  
* adopting an Event-Driven Architecture (EDA) with eventual consistency  
* selecting RabbitMQ as the event broker  
* adopting PostgreSQL as the primary relational database  
* the GreenGrant platform requiring storage for heterogeneous, high-volume, or rapidly evolving data  

---

## Facing the need for

* flexible schema support for evolving or semi-structured data (e.g., external manufacturer feeds)  
* efficient handling of high-volume ingestion data  
* storage of denormalized views optimized for fast reads (e.g., catalog projections)  
* the ability to scale horizontally for specific workloads  
* reducing complexity of mapping highly variable data into rigid relational schemas  

---

## We decided for

* adopting a NoSQL database (e.g., MongoDB) for specific use cases, with:

  * use limited to services that benefit from schema flexibility or high write throughput  
  * document-based storage for semi-structured or evolving data  
  * denormalized data models optimized for query patterns  
  * clear ownership of NoSQL data stores per microservice  
  * continued use of PostgreSQL for transactional and strongly consistent data  

---

## Achieving

* flexible handling of changing data structures without frequent schema migrations (adaptability)  
* improved performance for high-ingestion and read-heavy workloads (scalability)  
* simplified data models for specific use cases through denormalization (efficiency)  
* better alignment with event-driven projections and read models (CQRS support)  
* reduced impedance mismatch when dealing with external or inconsistent data formats  

---

## Accepting that (we will have)

* weaker consistency guarantees compared to relational databases  
* potential data duplication due to denormalization  
* the need to carefully design data access patterns upfront  
* additional operational overhead from managing multiple database technologies  
* increased complexity in data governance and backups  
* risk of misuse if NoSQL is applied to transactional use cases
