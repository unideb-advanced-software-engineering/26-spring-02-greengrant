---
title: "ADR-003: Adoption of RabbitMQ as Event Broker for GreenGrant"
description: "An architectural decision record describing the choice of RabbitMQ as the event broker for the GreenGrant system."
---

* Status: Active

## In the context of

* establishing a microservices architecture  
* adopting an Event-Driven Architecture (EDA) for inter-service communication  
* the need for a reliable messaging backbone to support asynchronous communication between services  
* the GreenGrant platform requiring event distribution for workflows such as ingestion, normalization, catalog updates, and notifications  

---

## Facing the need for

* a message broker that supports reliable asynchronous communication between services  
* flexible routing of events to multiple consumers  
* support for different messaging patterns (publish/subscribe, work queues, routing)  
* guaranteed message delivery with retry and dead-letter capabilities  
* manageable operational complexity for a growing but not yet hyperscale system  
* fast time-to-market with mature tooling and broad community support  

---

## We decided for

* using RabbitMQ as the primary event broker, with:

  * AMQP-based messaging for reliable delivery  
  * exchange types (direct, topic, fanout) for flexible routing  
  * queues for buffering and load leveling  
  * dead-letter queues for failure handling  
  * acknowledgments and retries to ensure message processing reliability  

---

## Achieving

* reliable asynchronous communication between services (resilience)  
* flexible routing of messages without tight coupling (modularity)  
* support for multiple messaging patterns within a single system (flexibility)  
* relatively simple setup and operation compared to high-throughput streaming platforms (operability)  
* fast adoption due to mature ecosystem, libraries, and documentation (productivity)  

---

## Accepting that (we will have)

* lower throughput and scalability compared to streaming platforms like Apache Kafka  
* limited support for long-term event storage and replay  
* the need to design around queue-based semantics rather than log-based streaming  
* potential bottlenecks under very high event volumes  
* operational responsibility for managing broker clustering, availability, and monitoring  
* possible future migration effort if system scale or analytics needs outgrow RabbitMQ
