---

title: "ADR-002: Adoption of Microservices Architecture for GreenGrant"
description: "An architectural decision record describing the choice of a microservices architecture for the GreenGrant system."
--------------------------------------------------------------------------------------------------------------------------------

* Status: Proposed

## In the context of

* the GreenGrant nationwide chocolate inventory platform, part of the Zamunda Digital Renaissance (ZDR)
* a system expected to grow in scale, complexity, and number of integrated external manufacturers and services
* the need to evolve different parts of the system (ingestion, processing, presentation, purchasing) at different speeds

---

## Facing the need for

* independent scaling of system components based on varying workloads (e.g., ingestion vs. user queries)
* faster and more flexible deployment cycles for different functional areas
* isolation of failures to prevent a single component from impacting the entire system
* support for multiple teams working in parallel on different domains
* clear separation of bounded contexts such as inventory ingestion, normalization, catalog management, and future purchasing features
* integration with heterogeneous external systems using different protocols and data formats
* long-term maintainability as system complexity increases

---

## We decided for

* a Microservices Architecture consisting of:

  * independently deployable services aligned with business capabilities (e.g., ingestion service, inventory service, notification service)
  * service-to-service communication via lightweight protocols (e.g., HTTP/REST and/or messaging)
  * decentralized data management, where each service owns its own data store
  * API-based interaction between services and external consumers

---

## Achieving

* independent scalability of services, allowing resource allocation based on actual demand patterns (scalability)
* improved fault isolation, ensuring that failures in one service do not cascade across the entire system (resilience)
* faster development and deployment cycles, enabling teams to release changes independently (agility)
* better alignment between system structure and business domains through bounded contexts (maintainability)
* flexibility in technology choices per service, enabling optimal tools for specific use cases (extensibility)
* easier integration of new features such as purchasing or analytics by introducing new services without major changes to existing ones

---

## Accepting that (we will have)

* increased architectural and operational complexity compared to a monolithic system
* the need for robust service discovery, monitoring, and distributed tracing
* challenges in ensuring data consistency across services (eventual consistency instead of strong consistency)
* more complex testing strategies, including integration and contract testing
* higher infrastructure costs due to multiple services and supporting components
* the need for well-defined API contracts and versioning strategies to avoid breaking changes
